# CLAUDE.md — プロジェクト基本ルール

## プロジェクト概要

AWS CLIを用いた作業手順書を作る、手順書生成システムを構築する。

## プロジェクトが解決したいこと

AWS CLI作業手順書（Runbook）の作成を効率化するツールキットの構築です。手順書はCloudShellで人間が読みながら対話的にコマンドを実行するもので、その運用自体は変えません。「書く側の労力」を減らすのが目的です。

### 仕様管理：SPEC.md
- 仕様はすべて SPEC.md に記載する
- 実装時は常に SPEC.md を常に参照し、仕様に従う
- 仕様の変更が必要な場合は、実装の前に SPEC.md を更新する

### タスク管理：TASKS.md
- タスクの計画と進捗状況は TASKS.md に記述する
- タスクが進行・完了したら、都度 TASKS.md を更新する
- ステータスの凡例：`[ ]` 未着手 / `[→]` 作業中 / `[✓]` 完了 / `[!]` レビュー待ち

### 作業ログ：LOGS.md
- 各作業の実施内容・決定事項は LOGS.md に追記する
- 1エントリは「日付見出し → 実施内容 → 決定事項」の構成で記述する
- 設計判断や仕様の経緯など、コードや差分から復元できない情報を残すことを目的とする

### 設計ナレッジ：KNOWLEDGE.md
- スニペット・ランブック開発のパターンと TIPS を記載している
- 新しいサービスやオペレーションを追加するときは、必ず KNOWLEDGE.md を参照して既存パターンに沿うこと
- 新しいパターンや判断基準が生まれた場合は KNOWLEDGE.md に追記する

### 設計の骨格（2層構造）：
- 手順書（Runbook） — 1リソース操作の手順（YAML定義 → テンプレートでMarkdown生成）
- スニペット — 1つのAWS CLIコマンドのテンプレート（Markdown断片）

## タスク作業フロー

1. developブランチから作業ブランチを作成する（例：`feature/generator`）
2. 成果物を作成する
3. TASKS.md のステータスを `[!]`（レビュー待ち）に更新する
4. ユーザーにレビューを依頼する
5. レビュー承認後、以下を実行する：
   - TASKS.md のステータスを `[✓]`（完了）に更新し、進捗サマリーも更新する
   - LOGS.md に作業ログを追記する
   - 変更をコミットする
   - developブランチに切り替えてマージする
   - 作業ブランチを削除する

## 成果物のルール

- 明示的に指定しない限り、テンプレートおよび成果物に絵文字は使用しない
